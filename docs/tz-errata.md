# Предложения правок в ТЗ → v2.3 (синхронизация с реализацией)

Код и repo-docs опередили ТЗ v2.2: ниже — конкретные пункты, которых не хватает
в документе для полного описания **уже реализованных** функций. Нумерация разделов —
по ТЗ v2.2. Детали каждого пункта — в `api-contract.md`, `data-model.md`, `decisions.md`.

## §9 REST API

1. **Дополнить таблицу эндпоинтов**: `GET /api/stores/:id` (публичная карточка),
   `GET|POST|PATCH|DELETE /api/stores/:id/products[/:pid]`. Указать модель доступа
   «Public с повышением прав» (анониму — `active`/`isPublic`, владельцу — всё).
2. **Добавить контракт ошибок**: единый конверт `{error:{code,message,details}}` и
   enum кодов `VALIDATION_ERROR, UNAUTHENTICATED, APP_CHECK_FAILED, FORBIDDEN,
   NOT_FOUND, CONFLICT, OUT_OF_STOCK, PROMO_INVALID, RATE_LIMITED, NOT_IMPLEMENTED,
   INTERNAL` с HTTP-кодами.
3. **Зафиксировать семантику partial-PATCH**: поле отсутствует → не изменяется;
   `""`/`null` для опциональных полей → очистка; производные поля (`totalStock`)
   пересчитывает сервер.

## §5–6 Функциональные требования

4. **Границы валидации** (сейчас нигде не нормированы): slug 3–40 +
   `^[a-z0-9]+(?:-[a-z0-9]+)*$`; имя магазина ≤120, товара ≤200; описания ≤2000/≤5000;
   SKU/штрихкод ≤64; категория ≤80; теги ≤20×40; фото ≤10; варианты ≤100;
   количество позиции корзины 1–99.
5. **FR-B04 детализация корзины**: ключ позиции `storeId+productId+variantKey`,
   фиксация цены на момент добавления (минорные единицы), помагазинность корзины.

## §4, §13 Аутентификация и безопасность

6. **Схема Custom Claims**: `{ storeId: string, role: 'owner' }` (+ `superadmin: true`);
   обязательный **force-refresh ID-токена** клиентом после `POST /stores/init`.
7. **App Check — режим тестируемости**: энфорсмент включён в `staging`/`production`,
   отключён в `development`/`test` (иначе невозможны эмуляторные интеграционные тесты
   и CI). Сейчас §13 требует App Check безусловно.
8. **Правило видимости**: посетителю — только товары `status='active'` и магазин
   `isPublic && !isBlocked`; владельцу — все статусы (отражено в Rules и API).

## §15 Инфраструктура

9. **Миниатюры**: конкретизировать Resize-extension — размеры 200×200 и 800×800,
   формат webp, соглашение имён `<имя>_<WxH>.webp`, путь `stores/*/products/**`,
   клиентский фолбэк на оригинал; download-token оригинала к миниатюре неприменим
   (публичное чтение по Rules).
10. **Чтение каталога**: п. «Экономия чтений Firestore» дополнить принятым решением —
    канонический путь чтения витрины: REST через сервер; прямое Firestore-чтение
    с офлайн-кэшем — оптимизация трека «Офлайн» (см. `decisions.md` №9).
11. **Версии стека и совместимость**: зафиксировать базовый тулчейн
    (Kotlin 2.1.0 / AGP 8.7.x / Compose BOM 2024.12 / **firebase-bom 33.x** —
    BOM 34.x требует Kotlin ≥2.3) и порядок его обновления.
12. **Окружения**: ТЗ требует Firebase-проекты dev/staging/prod — фактически на этапе
    разработки один проект (`wasat-21a20`); описать момент разделения (вместе с CD)
    либо скорректировать требование. Заменить плейсхолдер домена `app.example.com`
    реальным доменом для App Links / smart-link.

## §14 Тестирование
