# Принятые решения — экономическое обоснование (ТЗ §18)

По каждому ранее открытому вопросу выбран наиболее экономически эффективный вариант.
Решения отражены в соответствующих разделах документации.

| # | Вопрос | Принятое решение | Экономическое обоснование |
| --- | --- | --- | --- |
| 1 | Размещение Firestore | Единый регион `us-central1` (Iowa), Standard edition | Региональное размещение до ~50% дешевле мульти-региона; `us-central1` — недорогой регион; SLA 99.99% достаточно для MVP; задержка медиа закрывается CDN. Апгрейд на `nam5` — позже, при необходимости. |
| 2 | Веб-витрина | Только мобильное приложение + smart-link (без веб-витрины) | Параллельная веб-витрина удваивает разработку и поддержку. Публичные ссылки ведут на лёгкую redirect-страницу, открывающую приложение. SEO-поля товаров — в Post-MVP. |
| 3 | Тип Stripe Connect | **Express** | Stripe берёт на себя KYC, онбординг, комплаенс и дашборды выплат — минимум разработки и поддержки на платформе. |
| 4 | Комиссия платформы | Конфигурируемая; по умолчанию Free 5% / Basic 3% / Pro 2% / Enterprise 0–1% | Двойная монетизация: доход с транзакций на бесплатном тарифе, сниженная комиссия на платных как стимул к подписке. |
| 5 | Несколько магазинов на владельца | Один магазин на владельца в MVP | Исключает store-switcher и мульти-биллинг. Архитектура (UUID `storeId` + `ownerUid`) допускает N магазинов позже без переделки. |
| 6 | Стоимость Firestore при нагрузке | Паттерны экономии чтений | Разовые `get` для каталога (не listeners); realtime только для активного заказа; Firestore-бандлы + CDN; денормализованные агрегаты; пагинация. Валидация нагрузочным тестом. |
| 7 | Рекомендации (build vs buy) | MVP: эвристика in-house; ML — Post-MVP | «Недавно просмотренные» + «часто покупают вместе» на имеющихся данных заказов — нулевая доп. инфраструктура. Платный ML-сервис — только при подтверждённой отдаче. |
| 8 | AI и визуальный поиск | MVP: текстовый AI-ассист на экономичном LLM; визуальный поиск — Post-MVP | Текстовая генерация — низкая удельная стоимость с кэшированием. Визуальный поиск требует эмбеддингов и векторной БД (дорого) — откладывается. |

## Миниатюры изображений товаров (FR-A02)

Авто-генерация миниатюр — расширение **Storage Resize Images** (конфиг
`firebase/extensions/resize-product-images.env`, деплой:
`firebase deploy --only extensions`). Соглашение имён: рядом с оригиналом создаются
`<имя>_200x200.webp` и `<имя>_800x800.webp` (каталог/корзина используют 200x200,
карточка — 800x800). Клиент строит URL миниатюры по соглашению
(`core/util/ImageUrls.kt`) и откатывается на оригинал, если миниатюры ещё нет
(расширение не развёрнуто или ещё обрабатывает файл).

## Итоговая экономическая позиция

Решения минимизируют постоянные и инфраструктурные расходы MVP: один регион вместо
мульти-региона, отказ от второй (веб) платформы, перекладывание платёжного комплаенса
на Stripe, эвристические рекомендации и текстовый AI без дорогой инфраструктуры.
Дорогостоящие опции (мульти-регион, веб-витрина, ML-рекомендации, визуальный поиск)
сохранены как путь развития после подтверждения спроса.
