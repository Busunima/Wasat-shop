Документ фиксирует соответствие реализованного кода Техническому заданию
(«Android Multi-Store E-Commerce Platform», v2.2) и перечень оставшихся задач.

аудита кода (покупатель / владелец / суперадмин+инфраструктура) с привязкой к файлам.

(сохранённые карты Stripe) и **FR-S05** (биллинг Stripe), оба упираются в ключи Stripe;
плюс «дыры под внешние ключи» внутри в остальном готовых фич (живые платежи в FR-B05).
FR-A08/A10 — Post-MVP по самому ТЗ. **§11 (UI/UX) закрыт полностью** — Bottom Navigation,
haptics, skeleton-плейсхолдеры, navigation rail и list-detail на крупных экранах.
**Незаблокированный функционал ТЗ исчерпан.**

Обозначения: ✅ готово · 🟡 частично · 🚫 заблокировано внешней зависимостью ·

## Перечень оставшихся задач

### A. Заблокировано внешними решениями/ключами заказчика

1. **Stripe** — реализованы (env-gated, ждут живых ключей + деплой/Connect-настройку
   в Dashboard): приём оплат картой (B05) — серверный PaymentIntent + webhook +
   клиентский PaymentSheet; Connect-онбординг выплат (§10.2) — Express-аккаунт +
   `GET /stripe/onboard-link` + сохранение `stripeAccountId` + кнопка «Подключить
   выплаты» в настройках; **биллинг подписок S05 (сервер)** — subscription-checkout +
   webhook `/stripe-billing` (активация/grace/даунгрейд). Остаётся: сохранённые карты
   (B11), живой Refund (A11), клиентский выбор тарифа (S05), коллекция `invoices`,
   живые price ID тарифов.
2. **Algolia** — полнотекстовый/фасетный поиск (B02) + `POST /api/search/reindex`
   (отсутствует). Сейчас работает серверный fallback-поиск.
3. **SMTP/Nodemailer** — email-подтверждение заказа (B05), уведомление владельцу
   при блокировке (S02).
4. **Resize extension** — деплой `firebase deploy --only extensions` для миниатюр
   (A02); конфиг в репозитории присутствует.
5. **ANTHROPIC_API_KEY** — живой AI-ассист (A12).
6. **SHA-отпечатки + домен** — живой Google Sign-In и App Links
   (`https://app.example.com/s/{slug}`).
7. **Устройство** — Baseline Profiles (производительность, §11.8/§12).

### Сверка экранов витрины (2026-06-21)

Построчная вычитка каталога, карточки и редактора товара против §5/§6/§11 выявила
и закрыла (#86) расхождения, не отражённые в строках выше: **RangeSlider по цене**
был заплумблен в data-слое (`CatalogFilters`/`onPriceRangeChange`), но не имел
UI-контрола; язык AI-описания был жёстко `ru`; чипы вариантов обрезались до шести.
Дополнительно: в Категориях и Промокодах висел неиспользуемый `dismissError()` —
добавлена кнопка «Закрыть» баннера ошибки.

Сверка Корзины/Чекаута (FR-B04/FR-B05) выявила один незаблокированный UI-пробел:
кнопка «Оформить» оставалась активной офлайн (сбой ловился только после попытки),
хотя ТЗ требует блокировать чекаут до восстановления сети. Закрыто — `CheckoutViewModel`
потребляет уже существовавший `ConnectivityObserver` (раньше его читал только баннер
в `MainActivity`), кнопка гейтится по `online` + подсказка. Остальные пункты FR-B05
(Stripe PaymentSheet, Google Places, email-подтверждение, налоговая строка) — серверные
и/или ждут ключей заказчика, вне UI-объёма.

Сверка кабинета владельца (FR-A01/A03/A04/A05/A07/A11) выявила незаблокированные
UI-пробелы; закрыты чисто клиентские (сервер уже поддерживал контракт):
**FR-A05** — добавлены выбор периода (7/30/90д, сервер уже принимал `from`/`to`),
графики выручки по дням (рендер уже приходившего `daily`) и экспорт CSV (`AnalyticsCsv`
+ переиспользование `CsvShare`); **FR-A03** — кнопка «Повторить» в баннере ошибки
инвентаря (висел неиспользуемый `refresh()`).

**FR-A11** — отклонение возврата с причиной: оказалось клиентским, не серверным —
сервер уже принимал и хранил `comment` (`returnResolveSchema` → `resolveReturn`),
и outbox его прокидывал; не хватало только UI. Добавлен диалог причины при отклонении.

**FR-A04** — отмена заказа с причиной (client+server): `orderStatusUpdateSchema`
расширен полем `reason`, `updateOrderStatus` сохраняет `cancelReason` при CANCELLED
(под integration-тестом); клиент — диалог причины при отмене, прокидка через
outbox → API, отображение причины на карточке отменённого заказа.

**FR-A07** — сегменты и шаблоны рассылки: сегменты (client+server) — `broadcastSchema`
+= `segment`, `collectSegmentTokens` фильтрует адресатов по наличию `customerUid`
среди заказов (под integration-тестом); шаблоны (client-only) — заготовки, подставляющие
заголовок/текст. **На этом все находки аудита кабинета закрыты;** незаблокированный
функционал ТЗ (клиент + связанный сервер) исчерпан — остаток только за внешними
ключами/секретами заказчика (Stripe, Algolia, SMTP, Anthropic, Google Places, SHA).
