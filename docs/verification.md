# Верификация реализации против ТЗ v2.2

Документ фиксирует соответствие реализованного кода Техническому заданию
(«Android Multi-Store E-Commerce Platform», v2.2) и перечень оставшихся задач.
Дата сверки: 2026-06-13. Метод: построчная вычитка ТЗ (§1–§18) + три параллельных
аудита кода (покупатель / владелец / суперадмин+инфраструктура) с привязкой к файлам.

## Итог

Из **24 MVP-требований**: **~20 готовы (код-комплит)**, **3 частично**
(FR-B11, FR-A12, FR-S05), плюс «дыры под внешние ключи» внутри в остальном готовых
фич (живые платежи в FR-B05). FR-A08/A10 — Post-MVP по самому ТЗ.
Инфраструктура, безопасность (Firestore Rules + тесты, App Check, rate-limit, Zod),
CI/CD, локализация (ru/en), edge-to-edge и offline-first — реализованы.

Обозначения: ✅ готово · 🟡 частично · 🚫 заблокировано внешней зависимостью ·
⏸ Post-MVP (по ТЗ).

## Перечень оставшихся задач

### A. Заблокировано внешними решениями/ключами заказчика

1. **Stripe** — живой приём оплат (PaymentSheet/Google Pay, B05), сохранённые карты
   (B11), живой Refund (A11), Connect-онбординг + `onboard-link`,
   **биллинг подписок S05** (`/webhooks/stripe`, `/webhooks/stripe-billing`,
   dunning/grace), коллекции `invoices`/`subscription`. Каркас и env-gate готовы.
2. **Algolia** — полнотекстовый/фасетный поиск (B02) + `POST /api/search/reindex`
   (отсутствует). Сейчас работает серверный fallback-поиск.
3. **SMTP/Nodemailer** — email-подтверждение заказа (B05), уведомление владельцу
   при блокировке (S02).
4. **Resize extension** — деплой `firebase deploy --only extensions` для миниатюр
   (A02); конфиг в репозитории присутствует.
5. **ANTHROPIC_API_KEY** — живой AI-ассист (A12).
6. **SHA-отпечатки + домен** — живой Google Sign-In и App Links
   (`https://app.example.com/s/{slug}`).
7. **Устройство** — Baseline Profiles (производительность, §11.8/§12).

### B. Можно выполнять сейчас (не заблокировано)

1. **FR-A12 (довести до полного):** режим **rewrite** описаний + черновик
   SEO-мета + **Android-UI** кнопки «сгенерировать/переписать» в редакторе товара
   (серверный `generate`-эндпоинт уже есть).
2. **FR-B11 (без Stripe):** UI выбора адреса доставки в чекауте из адресной книги
   (несколько адресов; данные/повтор заказа уже есть).
3. **Центр уведомлений в приложении (§11.5)** — in-app список входящих push с
   историей (не реализован).
4. **Force-update через Remote Config (§11.5)** — min-version gate (не реализован).
5. **Offline-first хвост:** каталог-офлайн (Paging 3 RemoteMediator), WorkManager
   (фоновая синхронизация), outbox для возвратов/остатка, обработка протухшего
   токена офлайн.
6. **Тесты по §14:** E2E (Maestro) `регистрация→магазин→заказ`, load-тесты (k6)
   чекаут/поиск, E2E admin-web; perf-профайлинг.
7. **Модель данных §8:** при строгой сверке — вынести `stockNotifications` в
   коллекцию (сейчас массив в `customers/{uid}`; функционально работает).

### C. Post-MVP (зарезервировано самим ТЗ)

FR-A08 блог · FR-A10 лояльность · ML-рекомендации (Vertex AI / Recombee) ·
визуальный поиск · passkeys · мульти-регион Firestore (nam5).

## Реализованная инфраструктура и нефункциональные требования

- **Безопасность (§13):** App Check (env-gated), Firebase ID Token + Custom Claims,
  Firestore/Storage Rules (45+ тестов), rate-limit (Memorystore-путь задокументирован),
  двойная валидация (Kotlin + Zod), маскирование email в логах, auditLog + cron-очистка.
- **Тестирование (§14):** unit (ViewModel/Repository/Zod), integration (эмулятор
  Firestore), Rules-тесты, Android instrumentation, admin-web lint+build в CI.
  Пробелы: E2E (Maestro), load (k6), admin-web E2E.
- **Инфраструктура (§15):** Cloud Run, эмулятор-сьют, индексы, конфиг Resize
  extension, CI/CD (path-фильтры по компонентам). Пробелы: Baseline Profiles,
  Remote Config.
- **UI/UX (§11):** Material 3, edge-to-edge, предиктивный «назад», WindowSizeClass,
  локализация ru/en, a11y-подписи. Пробел: центр уведомлений в приложении.
- **Offline-first:** HTTP-кэш + офлайн-фолбэк + баннер сети; Room-кэш «моих заказов»,
  заказов магазина и карточки товара; outbox смены статуса заказа + индикатор
  синхронизации; серверная идемпотентность статусов/возвратов/остатка.
